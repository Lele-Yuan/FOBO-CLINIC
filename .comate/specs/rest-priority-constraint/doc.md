# 休息优先约束 — Rest Priority Constraint

## 问题分析

### 现象

如截图所示，在4人倒班场景下，白班人员（麦苗、煮蛋）出现了整周 `白(备) → 补 → 白(备) → 补 ...` 的交替模式：

- 麦苗：补 / 白(备) / 补 / 白(备) / 补 / 白(备) / 补
- 煮蛋：白(备) / 补 / 白(备) / 补 / 白(备) / 补 / 白(备)

结果：全周无一天真正的 `休`，实际休息 = 0 天，严重违背 T_week = 1.5 天的约束。

### 根本原因

`pass2Standby()` 分配备班时**不检查该人员本周是否仍有足够余量达到 T_week**：

- 当白班人员池较小（仅2人），备班轮询指针会在它们之间高频轮转
- 每次备班占用当天（白备）+ 次日（补），两天全部锁定
- `pass4WeeklyRest()` 执行时，这些人在该周已无空闲 `白` 班可转为 `休`
- 虽然 `'补'` 已正确排除在 `_calcRestScore()` 计数外，但此时已无日期可补偿

### 两条最高优先级规则

> **任何情况下都不能违背：**
>
> 1. **每周休息达标**：每人每自然周（Mon-Sun）内，非 `补` 类型的休息时间之和 ≥ T_week（默认 1.5 天）
> 2. **补休不计入每周休息额度**：`'补'` 班次只是备班后的晨休补偿，不算作正式休息天数

## 技术方案

### 核心改动：在 pass2Standby 中加入周休可行性预检

**文件：** `pages/schedule/schedule.js`

在 `pass2Standby()` 选人逻辑（LINE ~180）中，为每位候选人调用新增的 `_canAssignStandby()` 函数进行预检。

### _canAssignStandby(schedule, date, personId, config) 逻辑

```
输入：
  - schedule: 全局排班表
  - date: 备班日期
  - personId: 候选人ID
  - config: 配置（T_week、forcedWorkDays等）

算法：
1. 获取 date 所在自然周的所有日期 weekDays（周一至周日）
2. 遍历 weekDays，对每个日期 d 的 cell：
   a. 已固定的休息 = 已是 '休'/'下'(+1.0) 或 '上半'(+0.5)，不含 '补'
   b. 今日 date → 将变为 '白(备)'，不计休息，不再视为自由日
   c. 次日 date+1 → 将变为 '补'，不计休息（若 date+1 在本周内）
   d. 其余未固定的 '白' 班（非 manual、非 standby、非已排休）→ 视为可用自由日（+1.0）
3. 计算：可达休息 = 已固定休息 + 可用自由日数
4. 判断：可达休息 >= T_week → 可分配；否则跳过此人
```

### 边界条件

- **跨周次日**：若 `date` 为周日（Sunday），次日 `date+1` 属于下一周，不影响本周可行性检查，本周仅损失今天
- **半天处理**：T_week=1.5 时，可用自由日可以是1天休+1个半天，自由日估算按全天（1.0）保守计算
- **手动排班**：`cell.manual === true` 的日期不可被 pass4 修改，视为不可用自由日
- **已有夜班**：rotate 人员在本周若有 `'下'`（夜班后全休）已计入固定休息
- **forcedWorkDays**：若某天是强制上班日（Mon/Wed/Fri），pass4 不会在这些天排休，这些天不应计为自由日 → 从可用自由日中排除

### 数据流

```
buildGlobalSchedule()
  └─ pass1Base()           # 设置基础班次
  └─ pass3NightAfterRest() # 夜班次日强制下
  └─ pass2Standby()        # 备班分配 ← 加入 _canAssignStandby() 预检
       └─ _canAssignStandby()  # 新增：周休可行性检查（最高优先级）
       └─ _setAMRest()         # 设置次日补休
  └─ pass4WeeklyRest()     # 填充每周休息额度（此时已有足够空间）
```

## 预期效果

修改后，白班人员每周必然有足够空闲日用于 pass4 分配真正的 `休` 天：

- 麦苗/煮蛋 这类白班人员不会被过度分配备班
- 当备班候选池过小时，系统优先保障休息，可能导致某天没有备班（备班需求无法完全满足时，休息优先）
- `补` 依旧不计入周休额度（规则2不变）

---

## 已排除方案

- **调整 pass4 补偿**：pass4 执行时已无空余日期，无法修复
- **减少备班需求**：用户需求固定，不适合降低配置
- **增加白班人员**：由配置决定，算法层面不控制
